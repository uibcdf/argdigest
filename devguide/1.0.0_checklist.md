# 1.0.0 Stability and Ecosystem Checklist

This checklist merges ArgDigest's internal release path with ecosystem-level
requirements observed in `../molsyssuite/devguide`.
It is the operational gate for shipping `1.0.0` without ambiguity.

## Status Legend

- `[x]` complete and verified
- `[ ]` pending
- `[-]` intentionally deferred to post-1.0

## A) Core 1.0 must-haves (release blocking)

- [x] freeze and verify precedence for configuration discovery (`runtime > env > _argdigest.py`).
- [x] keep `arg_digest` behavior stable across package, registry, decorator, and mixed styles.
- [x] guarantee deterministic digester discovery and execution order.
- [x] complete API contract lock for public decorators, errors, and pipelines.
- [x] verify docs/API/runtime consistency for all exported public symbols.

Current evidence:
- style stability verified by smoke and contract tests (`tests/test_smoke_examples.py`, `tests/test_argument_digestion.py`).
- API lock verified (`tests/test_api_contract.py`).
- docs/API/runtime export consistency verified (`tests/test_api_reference_contract.py`).
- precedence with environment verified (`tests/test_config_precedence_env.py`) and documented in user/developer docs.
- deterministic package digester load order enforced and contract-tested (`tests/test_argument_loader_order.py`).

## B) Cross-library interoperability must-haves (release blocking)

- [x] document and enforce compatibility matrix with sibling libraries (`smonitor`, `depdigest`, `pyunitwizard`).
- [x] ensure all critical public API paths emit coherent SMonitor signals.
- [x] ensure ArgDigest diagnostics include stable `CODE` and machine-readable `hint`.
- [x] verify optional dependency behavior never hard-requires heavy stacks without DepDigest-compatible flow.

Current evidence:
- compatibility matrix documented (`docs/content/developer/compatibility-matrix.md`) and version floors enforced (`pyproject.toml`, `tests/test_compatibility_matrix.py`).
- SMonitor instrumentation verified on critical execution paths (`argdigest.core.decorator` wrapper, `Registry.run`, `get_pipelines`) in `tests/test_interop_block_b.py`.
- catalog-backed exceptions/warnings expose stable `code` and `hint` fields (`tests/test_interop_block_b.py`).
- optional contrib modules import without hard optional dependencies (`tests/test_interop_block_b.py`).

## C) Validation semantics and UX must-haves (release blocking)

- [x] ensure contract failures are friendly and concise in `user` profile.
- [x] ensure `dev` profile exposes full digestion trace with actionable context.
- [x] map `pyunitwizard` conversion failures into ArgDigest contract errors with caller context.
- [x] validate strictness modes (`raise`, `warn`, `silent`) against documented behavior.
- [x] verify warning/error taxonomy is consistent across registry and package styles.

Current evidence:
- profile-specific message behavior validated for `user` vs `dev` (`tests/test_block_c_validation_semantics.py`).
- pyunitwizard conversion failures mapped to `DigestValueError` with preserved caller/argument context (`tests/test_block_c_validation_semantics.py`, `tests/test_low_coverage_modules.py`).
- strictness aliases normalized and validated (`raise/error`, `warn/warning`, `silent/ignore/none`) in runtime + docs (`argdigest/core/decorator.py`, `docs/content/user/strictness-and-errors.md`, `tests/test_block_c_validation_semantics.py`).
- warning taxonomy parity verified across package and registry styles (`tests/test_block_c_validation_semantics.py`).

## D) Performance and reliability must-haves (release blocking)

- [x] validate `inspect.signature` caching strategy under repeated high-frequency calls.
- [x] verify no regression in decorator overhead versus current baseline.
- [x] verify import-time behavior remains lightweight and deterministic.
- [x] ensure thread-safe/read-safe behavior for caches and registries in common usage patterns.

Current evidence:
- metadata cache reuse for digesters validated under repeated calls (`tests/test_block_d_performance_reliability.py`).
- decorator overhead bound checked in a hot-loop baseline scenario (`tests/test_block_d_performance_reliability.py`).
- import-time behavior verified in subprocess (lightweight import and no optional heavy stack imports) (`tests/test_block_d_performance_reliability.py`).
- thread-safe/read-safe behavior validated for registry and digestion paths; locks added for metadata cache and registries (`argdigest/core/decorator.py`, `argdigest/core/registry.py`, `argdigest/core/argument_registry.py`, `tests/test_block_d_performance_reliability.py`).

## E) Tooling and release gates must-haves (release blocking)

- [ ] CI workflow green (tests, lint, coverage threshold).
- [ ] docs workflow green and `gh-pages` publication successful.
- [ ] wheel install/import check fails fast if package import is broken.
- [ ] conda build/upload workflow validated for current release tag.
- [ ] release notes and migration notes finalized for `1.0.0`.
- [x] validate ecosystem health check flow (SMonitor/ArgDigest/DepDigest/PyUnitWizard) in an integration scenario.

Current evidence:
- ecosystem health-check flow implemented via `argdigest health-check` and validated by tests (`argdigest/core/health.py`, `tests/test_health_check.py`).
- CI/docs/conda gates remain release-managed and must be confirmed on tagged workflows before `1.0.0`.

## F) Cross-layer end-to-end criterion (release blocking)

- [ ] validate that a `pyunitwizard` unit failure is re-emitted by ArgDigest as a contract error with caller context.
- [ ] validate that the emitted ArgDigest diagnostic includes stable `CODE` and machine-readable `hint`.
- [ ] validate dependency remediation hint path (`depdigest`) when failure cause is missing optional stack.
- [ ] confirm this cross-layer behavior is coherent in `user` and `dev` profiles.

## G) Strong should-haves (target for 1.0 if feasible)

- [ ] add explicit integration confidence tests against MolSysMT representative paths.
- [ ] provide at least one end-to-end example per digestion style in docs/showcase.
- [ ] add negative-path tests for malformed `_argdigest.py` and invalid registry wiring.
- [ ] provide an explicit troubleshooting page for end users receiving ArgDigest/SMonitor messages.

## H) Deferred research and future vision (non-blocking for 1.0)

- [-] adaptive validation throttling (trust-based runtime validation reduction).
- [-] autonomous or AI-assisted semantic coercion for ambiguous argument inputs.
- [-] autonomous fuzzing generation from digestion contracts.
- [-] dynamic/JIT dependency provisioning during runtime.
- [-] counterfactual triage and self-healing workflows driven by runtime diagnostics.

## Final release criterion

ArgDigest is ready for `1.0.0` when sections A-F are fully checked and no
open blocker remains in digestion core, configuration resolution, diagnostics,
or ecosystem compatibility.
